### 2. ปรับปรุงกระบวนการ Sync ✅

#### 2.1 บุคลากร (staff_info) — จาก MySQL ต้นทาง

**ปัญหาเดิม:**
- UPSERT ทีละ row ไม่ลบ record เก่า → เกิด "Ghost Records" (1,253 records ทั้งที่ active จริง 1,089)
- บุคลากรที่ลาออกหรือย้ายแล้วยังค้างในระบบ → ส่งผลต่อระบบ SSO

**แนวทางแก้ไข:** UPSERT batch 500 + DELETE stale records

```
ก่อน: ~1,089 INSERT/UPDATE queries
หลัง: ~3 queries (executemany batch + DELETE stale)
เวลา: 0.4s
```

#### 2.2 นักศึกษา (students_info) — จาก Oracle DB

**ปัญหาเดิม:**
- Row-by-row: SELECT EXISTS → UPDATE หรือ INSERT สำหรับ 14,000+ records
- ใช้เวลาหลายนาที, ไม่ atomic

**แนวทางแก้ไข:** DELETE all + bulk INSERT in batches (InnoDB transaction เดียว)

```
ก่อน: 28,000+ queries (SELECT + UPDATE/INSERT ทุก row)
หลัง: ~23 queries (1 DELETE + batch INSERTs)
เวลา: 2.5s (จากเดิมหลายนาที)
```

**ผลลัพธ์ทดสอบบน production:**
| ตาราง | Records | เวลา |
|---|---|---|
| staff_info | 1,089 | 0.4s |
| students_info | 11,280 | 2.5s |

### 3. Oracle Thick Mode ✅

**ปัญหา:** Oracle server เก่า (ก่อน 12.1) → oracledb thin mode ไม่รองรับ (DPY-3010 error)

**แก้ไข:**
1. ติดตั้ง Oracle Instant Client 21.20 ที่ `C:\oracle\instantclient_21_20\`
2. เพิ่ม `.env`: `ORACLE_CLIENT_LIB=C:/oracle/instantclient_21_20`
   > ⚠️ ต้องใช้ forward slash — backslash ทำให้ env var อ่านไม่ได้

### 7. ย้ายจาก APScheduler → Windows Task Scheduler ✅

**ปัญหาเดิม:**
- APScheduler รันอยู่ภายใน Django process → ถ้า service crash ต้อง restart เพื่อให้ scheduler กลับมา
- Bug: `sys.argv` exact match ไม่ match `deploy/waitress_serve.py` → scheduler ไม่เคย start บน production เลย

**แก้ไข:**
1. ปิด APScheduler ใน `apps.py`
2. สร้าง `deploy/task_scheduler/sync_staff.bat` และ `sync_students.bat`
3. สร้าง `deploy/task_scheduler/register_tasks.ps1` สำหรับลงทะเบียน tasks
4. เพิ่ม `--triggered-by` argument ใน management commands → Sync Monitor แสดง "auto"

**ผลการทดสอบ:**
- รัน bat ด้วยมือ: `exit code: 0` ทั้ง staff และ students
- Task Scheduler trigger ผ่าน `Start-ScheduledTask`: ทำงานสมบูรณ์
